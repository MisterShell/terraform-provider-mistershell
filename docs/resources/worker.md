# mistershell_worker (Resource)

Manages a MisterShell worker — a task-handler agent bound to a location. The config and config_schema attributes are opaque JSON blobs (the worker's task-handler config and its JSON Schema). The bootstrap default worker (is_default=true) is provisioned by the backend and cannot be edited or deleted (the API returns 403); do not manage it with this resource. The authentication token is returned only once at creation and is never readable again.

## Example Usage

```terraform
# A worker hanging under a location, with task-handler config and a config schema.
resource "mistershell_location" "emea" {
  name      = "EMEA"
  parent_id = 1
}

resource "mistershell_worker" "example" {
  name        = "emea-worker-1"
  description = "Worker serving the EMEA location"
  location_id = mistershell_location.emea.id
  is_enabled  = true

  # Opaque, free-form task-handler config pushed to the worker via heartbeat.
  # Keys are read by task handlers with get_config_value(key, default); the only
  # backend-recognized key today is manual_worker_presence_ip (admin override for
  # the IP shown in the UI). Everything else is consumed by your task handlers.
  config = jsonencode({
    manual_worker_presence_ip = "203.0.113.10"
  })

  # Optional JSON Schema the backend uses to validate the config above.
  config_schema = jsonencode({
    type = "object"
    properties = {
      manual_worker_presence_ip = { type = "string" }
    }
    additionalProperties = true
  })
}

# The authentication token is returned ONLY at creation time and can never be
# read back afterwards. It is stored in Terraform state (sensitive). If lost,
# regenerate it out of band (POST /workers/{id}/regenerate-token).
output "worker_token" {
  value     = mistershell_worker.example.token
  sensitive = true
}
```

### `status` values

`status` is read-only and tracks the worker's **operational** state (what is
actually happening), independent of the `is_enabled` admin intent.

| status | meaning |
|---|---|
| `pending` | Created but has never connected to the broker yet. The initial state of every newly created worker. |
| `online` | Connected to the broker and accepting tasks. |
| `offline` | Was connected but is now disconnected (no recent heartbeat). |
| `error` | In a connection error state. |

## The default worker

MisterShell bootstraps a single **default worker** (`is_default = true`) at
startup for the all-in-one container, bound to the root location. This worker is
system-managed:

- It **cannot be created** via the API (`POST /workers` with `is_default = true`
  returns **403**), so you cannot reproduce it with this resource.
- It **cannot be edited or deleted** (PATCH / DELETE return **403**).

Do **not** manage the default worker with `mistershell_worker`. Use this resource
only for additional, user-created workers.

## The `token`

`token` is the worker's authentication token (format `wkr_<id>_<random>`). It is:

- **Returned only once**, in the creation response. The server stores only a hash
  and never returns the token again — not on `Read`, the data source, or import.
- **Stored in Terraform state** and marked **sensitive**. Treat your state as a
  secret. Surface it with a `sensitive = true` output (see the example) if you
  need to capture it.
- **Not populated on import.** After importing a worker, `token` is empty in
  state and cannot be recovered from the server.

If the token is lost, **regenerate it out of band** with
`POST /workers/{id}/regenerate-token`; the old token is invalidated immediately.

## `config` and `config_schema`

Both are opaque JSON blobs written with `jsonencode(...)` and sent to the backend
as JSON objects.

- **`config`** is the worker's free-form **task-handler configuration**. The
  backend does not impose a fixed shape: keys are pushed to the worker via the
  heartbeat and read by task handlers as needed (`get_config_value(key, default)`).
  The only key the backend itself interprets today is
  `manual_worker_presence_ip` (a string IP that overrides the auto-detected
  `presence_ip`, surfacing `presence_ip_source = "manual"`); every other key is
  consumed by your task handlers. Because it is free-form, **document your own
  keys** for your deployment. A representative example:

  ```hcl
  config = jsonencode({
    manual_worker_presence_ip = "203.0.113.10" # optional admin override for presence_ip
    # ...any additional keys your task handlers read...
  })
  ```

- **`config_schema`** is an optional **JSON Schema** the backend uses to validate
  `config`. It too is free-form JSON:

  ```hcl
  config_schema = jsonencode({
    type = "object"
    properties = {
      manual_worker_presence_ip = { type = "string" }
    }
    additionalProperties = true
  })
  ```

**Stored-from-config.** As with `credential_data` and log-destination `config`,
the provider stores `config` and `config_schema` **from your configuration** and
does not read them back from the server (the server normalizes/reorders keys and
derives `config_version`). Two consequences:

- **Out-of-band, non-secret changes are not detected as drift** — the whole blob
  is preserved from state. Manage these values exclusively through Terraform.
- Both are **excluded from import** (only `id` is set on import); re-supply them
  in HCL after importing, exactly like `credential_data`.

## Import

Import is supported using the following syntax:

```shell
# Workers are imported by their integer ID.
# Note: token, config, and config_schema are NOT read back on import. The token
# can never be recovered from the server (regenerate it out of band if lost),
# and config / config_schema are stored-from-config — re-supply all three in
# your configuration after importing.
terraform import mistershell_worker.example 123
```
