# mistershell_auth_provider (Resource)

Manages a MisterShell external authentication provider (LDAP, OIDC, or SAML). The config attribute is an opaque per-provider_type JSON blob carrying secrets; the API masks secrets ('****') and enriches non-secret defaults, so config is stored from your plan (not read back). display_order is set declaratively (honored on create; omitted appends at the end). Create and update require an active license.

## Example Usage

```terraform
# An OIDC auth provider. config is an opaque per-provider_type JSON blob set
# with jsonencode(); client_secret is a secret (masked '****' by the API and
# stored from your configuration, not read back). display_order is set
# declaratively via a post-create PATCH; assign distinct values for a
# deterministic login-page order.
#
# Create and update require an active license.
resource "mistershell_auth_provider" "corp_oidc" {
  name          = "corp-oidc"
  provider_type = "OIDC"
  is_enabled    = true
  display_order = 0

  config = jsonencode({
    issuer_url    = "https://login.example.com"
    client_id     = "mistershell"
    client_secret = var.oidc_client_secret # secret
    scopes        = ["openid", "profile", "email"]
  })
}

variable "oidc_client_secret" {
  type      = string
  sensitive = true
}
```

## `display_order` semantics

`display_order` controls the login-page ordering of providers:

- **Omit `display_order`** to keep the server's auto-assigned value (`max + 1` on
  create). The value is read back into state (Optional + Computed), so omitting it
  does not produce a perpetual diff.
- **Assign distinct values** for a deterministic login-page order. The backend
  does **not** enforce unique `display_order`; duplicates break ties by `id`.
- **Create cannot set `display_order`** (the server auto-assigns it). When you pin
  a value, the provider applies it with a follow-up PATCH after create; thereafter
  it is set on update. The provider **never** calls the backend's reorder endpoint.

## The `config` JSON blob

`config` is an opaque, per-`provider_type` JSON object written with
`jsonencode(...)`. Its shape is determined by `provider_type` (one table per type
below). Each type carries at least one **secret** field.

**Secret & drift handling.** Every secret field (LDAP `bind_password`, OIDC
`client_secret`, SAML `sp_private_key`) is encrypted at rest and **masked to
`****`** in the API's responses. As with `credential_data`, the provider stores
the **entire `config` from your configuration** and does **not** read it back from
the server — for **all** provider types, not just the secret fields. This is
necessary because the server also **enriches** non-secret defaults (LDAP
`user_search_filter`/`user_attributes`; OIDC `scopes`/`claims_mapping`/
`redirect_uri`; SAML `sp_entity_id`/`sp_acs_url`/`attribute_mapping`), so
reflecting the server value would churn. Consequences:

- After **importing** a provider, `config` is unknown until you re-supply it in
  HCL — secrets cannot be recovered from the masked server value. `config` is
  **excluded from import** (only `id` is set on import).
- Out-of-band changes to the **non-secret** parts of `config` are **not** detected
  as drift, because the whole `config` is preserved from state. This is the
  documented trade-off of the stored-from-config rule (identical to
  `credential_data`).

**License.** `config` create/update (and group-mapping create/update) require an
active license; an unlicensed instance returns a `403` error, surfaced verbatim.
Delete works unlicensed.

## Import

Import is supported using the following syntax:

```shell
# Auth providers are imported by their integer ID.
# Note: config is NOT read back from the server (the API masks secrets and
# enriches non-secret defaults); after importing, set config in your
# configuration yourself. config is excluded from import.
terraform import mistershell_auth_provider.corp_oidc 3
```
