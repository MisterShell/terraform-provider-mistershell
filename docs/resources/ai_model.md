# mistershell_ai_model (Resource)

Manages a MisterShell AI model registration, pairing a provider and provider-specific model id with an opaque per-provider config blob (api_key, base_url, etc.). The config attribute is sensitive: provider secrets are masked by the API and stored from config (not read back).

## Example Usage

```terraform
# Anthropic model using an API key supplied from a sensitive variable.
# This model is marked as the default for agents that do not pin a model.
resource "mistershell_ai_model" "claude" {
  name           = "claude-sonnet"
  model_provider = "anthropic"
  model_id       = "claude-3-5-sonnet-latest"
  is_default     = true

  config = jsonencode({
    api_key = var.anthropic_api_key # secret; masked as "***" by the API, stored from config
  })
}

# Ollama model running locally — OpenAI-compatible API, no API key required.
# Only a base_url is needed (defaults to http://localhost:11434/v1 server-side).
resource "mistershell_ai_model" "ollama_llama" {
  name           = "local-llama"
  model_provider = "ollama"
  model_id       = "llama3.1"

  config = jsonencode({
    base_url = "http://localhost:11434/v1"
  })
}

# OpenAI model pointed at a custom gateway via base_url.
resource "mistershell_ai_model" "gpt" {
  name           = "gpt-4o"
  model_provider = "openai"
  model_id       = "gpt-4o"

  config = jsonencode({
    api_key  = var.openai_api_key # secret
    base_url = "https://gateway.example.com/v1"
  })
}

variable "anthropic_api_key" {
  type      = string
  sensitive = true
}

variable "openai_api_key" {
  type      = string
  sensitive = true
}
```

## Schema

### Required

- `model_id` (String) Provider-specific model identifier (e.g. "claude-3-opus", "gpt-4o").
- `model_provider` (String) AI provider backing this model. One of: anthropic, azure_openai, bedrock, cohere, google, mistral, ollama, openai, openrouter, xai. (Named `model_provider` because `provider` is a reserved Terraform argument.)
- `name` (String) AI model name (1-255 characters).

### Optional

- `config` (String, Sensitive) Per-provider config as JSON. Use jsonencode() in HCL. Common keys: api_key, base_url, context_window, api_version, region, deployment. Secrets such as api_key are write-only: they are masked as '***' by the API on read and stored from config, not read back. See the config field tables on this page.
- `is_default` (Boolean) Whether this is the default AI model. Defaults to false. Only one model may be the default: setting this to true clears the default flag on all other models. Manage is_default from a single resource to avoid drift.

### Read-Only

- `created_at` (String) Creation timestamp.
- `id` (Number) AI model ID.
- `updated_at` (String) Last update timestamp.

### `is_default` behaviour

`is_default` selects the model used by agents that do not pin a specific model.
**At most one model can be default.** Setting `is_default = true` on a model
clears the flag on every other model **server-side** in the same operation.

To avoid perpetual drift, manage the default flag from a **single** resource. If
two `mistershell_ai_model` resources both set `is_default = true`, whichever is
applied last wins and the other will show drift on the next plan.

## The `config` JSON blob

`config` is an opaque JSON object written with `jsonencode(...)`. The backend
stores it as free-form JSON and **passes it through to the provider SDK** — it
does not validate the key set, so any key the underlying provider accepts can be
supplied. In practice only a small, well-known set of keys is consumed by
MisterShell when it builds the model client.

**Keys consumed by MisterShell** (everything else is stored but ignored by the
model-construction path; per-request generation settings such as `max_tokens`,
`temperature`, and `top_p` come from the **agent** config, not the model config):

| key | type | applies to | secret | notes |
|---|---|---|---|---|
| `api_key` | string | all except `ollama` | **yes** | Provider API key. Required for all providers except `ollama` (which accepts none and defaults to the literal `ollama`). Masked `***` in reads; stored from config. |
| `base_url` | string | `anthropic`, `openai`, `ollama`, `bedrock` (custom endpoints) | no | Custom endpoint. For `ollama` defaults server-side to `http://localhost:11434/v1`. Used for OpenAI/Anthropic-compatible gateways and Bedrock custom endpoints. |
| `model_id` | string | all | no | Optional override of the top-level `model_id` when building the client. Normally leave unset and use the top-level `model_id` attribute. |

**Secret handling** (identical to `credential_data` / log-destination `config`):

- The backend masks any config key whose name (case-insensitive) is one of
  `api_key`, `secret`, `secret_key`, `token`, `password`, `access_key` to `***`
  in every API response.
- The provider stores `config` **from your configuration** and does **not** read
  it back from the server. Consequences:
  - After **importing** a model, its secrets are unknown until you re-supply the
    full `config` in HCL — a masked `***` value cannot be recovered.
  - Out-of-band changes to the **non-secret** parts of `config` are **not**
    detected as drift, because the whole `config` is preserved from state.
  - `config` is **excluded from import** (only `id` is set on import); re-supply
    it in HCL, exactly like `credential_data`.
- On update, if you omit `api_key` from `config`, the server **preserves** the
  previously stored `api_key`. To rotate the key, supply the new value
  explicitly.

### Per-provider config

The provider passes `config` through to the Pydantic AI provider SDK, so the
authoritative key set is the SDK's. The keys MisterShell itself reads when
constructing the client are `api_key`, `base_url`, and the optional `model_id`
override (see table above). The minimal/typical config per provider:

| model_provider | typical config |
|---|---|
| `anthropic` | `{ api_key = "..." }` (+ optional `base_url`) |
| `openai` | `{ api_key = "..." }` (+ optional `base_url` for a compatible gateway) |
| `azure_openai` | `{ api_key = "..." }` (+ provider-SDK keys such as `base_url`/endpoint, `api_version`, `deployment` as required by your Azure resource) |
| `bedrock` | `{ api_key = "..." }` (+ optional `base_url`; AWS credential/region settings as required by your environment) |
| `google` | `{ api_key = "..." }` |
| `mistral` | `{ api_key = "..." }` |
| `cohere` | `{ api_key = "..." }` |
| `xai` | `{ api_key = "..." }` |
| `openrouter` | `{ api_key = "..." }` |
| `ollama` | `{ base_url = "http://localhost:11434/v1" }` — no `api_key` (OpenAI-compatible; key defaults to `ollama`) |

Examples:

```hcl
# Anthropic — api_key only
config = jsonencode({
  api_key = var.anthropic_api_key # secret; masked "***", stored from config
})

# Ollama — base_url only, no key
config = jsonencode({
  base_url = "http://localhost:11434/v1"
})

# OpenAI behind a custom gateway
config = jsonencode({
  api_key  = var.openai_api_key # secret
  base_url = "https://gateway.example.com/v1"
})
```

## Deletion

Deleting a model that is **referenced by one or more agents** is rejected by the
backend with HTTP **409 Conflict** (`Cannot delete model: it is referenced by
one or more agents`). Remove or repoint the referencing agents first, then
delete the model.

## Import

Import is supported using the following syntax:

```shell
# AI models are imported by their integer ID.
# Note: config secrets (api_key and any key named secret/secret_key/token/
# password/access_key) are NOT read back from the server (the API masks them as
# "***"); re-supply the full config in your configuration after importing.
terraform import mistershell_ai_model.example 123
```
