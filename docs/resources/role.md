# mistershell_role (Resource)

Manages a MisterShell role. A role groups a set of permission names and an optional location scope. Permission names are validated by the backend (no static enum). The superuser permission '*.*.*' cannot be combined with other permissions or with a location scope.

## Example Usage

```terraform
# A read-only auditor role scoped to two locations.
# scope_location_ids limits the role to the given location subtrees;
# omit it (or leave it empty) to grant the role across all locations.
resource "mistershell_role" "auditor" {
  name        = "auditor"
  description = "Read-only access to tags and resources in EMEA"

  scope_location_ids = [
    mistershell_location.emea.id,
    mistershell_location.apac.id,
  ]

  permissions = [
    "app.tags.read",
    "app.resources.read",
  ]
}

# A global superuser role. The '*.*.*' permission cannot be combined with any
# other permission, and cannot be used together with scope_location_ids.
resource "mistershell_role" "superuser" {
  name        = "superuser"
  description = "Full administrative access"

  permissions = ["*.*.*"]
}
```

### `scope_location_ids`

`scope_location_ids` is the set of location IDs that scope the role: the role's
permissions apply only within those location subtrees. An **empty set or an
omitted attribute means the role is global** (applies to all locations). It is a
Set (order-independent); IDs must reference existing locations.

The superuser permission `*.*.*` **cannot** be combined with a non-empty
`scope_location_ids` — a superuser is global by definition. Setting both is
rejected by the server.

### `permissions`

`permissions` is the set of permission names assigned to the role. Names follow
the pattern `module.resource.action` (for example `app.tags.read`), plus the
single wildcard superuser permission `*.*.*`.

Permission names are **validated server-side**, not by a static enum in the
provider. The authoritative, always-current list of valid names is published by
the product itself through the [`mistershell_permissions`](../data-sources/permissions.md)
data source — query it for the live per-module enumeration. The provider does
not hardcode the list because permission names live in a server-side registry
that is not part of the published OpenAPI schema, so the on-product data source
is the single source of truth; an unknown name is surfaced as a server error.

#### Superuser rules

The `*.*.*` permission grants everything and is mutually exclusive:

- `*.*.*` **cannot** be combined with any other permission name.
- `*.*.*` **cannot** be used together with `scope_location_ids` (it is always
  global).
- No additional permission can be added to a role that already holds `*.*.*`.

All of these are enforced by the server and reported as an error on apply.
